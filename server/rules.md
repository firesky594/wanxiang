# Wanxiang Agent 后端规范

本规范适用于 `server/` 下全部 Go 源码、测试、数据库访问、后台任务和部署操作。
用户最新明确要求优先于本文。

## 1. 目录与职责

- `cmd/wanxiang/`：仅负责解析启动参数、加载配置、创建应用和监听 HTTP。
- `internal/app/`：统一装配数据库、Service、Worker、事件总线和 HTTP 依赖。
- `internal/httpapi/`：维护路由、鉴权、中间件和输入输出转换，Handler 不复制业务逻辑。
- `internal/<domain>/`：按领域维护 Service、Worker、类型和持久化逻辑，禁止页面接口层直接跨域写表。
- `internal/db/`：统一管理 SQLite 连接、连接参数和可重复执行的迁移。
- `internal/testutil/`：只保存可复用测试辅助代码，不得被生产业务依赖。
- `test/`：后端测试源、测试运行器和测试说明的唯一归档目录。
- `server_func_doc.md`：后端可跨文件调用的项目自定义导出函数与方法索引。

## 2. 函数、注释与复用

- 项目自定义的导出函数和导出方法必须有中文 Go Doc 注释，说明职责而不是复述函数名。
- 新增或修改可复用导出函数时，必须同步更新 `server_func_doc.md`。
- `server_func_doc.md` 只记录项目源码中的导出函数和方法，不记录 Go 标准库、Chi、
  SQLite 驱动或其他依赖提供的方法。
- 同一路径、接收者和方法名只能登记一次；相同业务能力必须统一调用，禁止复制实现。
- 包内一次性闭包、匿名回调、测试辅助函数、简单字段访问器不进入函数索引。
- 注释、函数索引和实际行为必须一致，行为变化时三者同时更新。

## 3. HTTP、鉴权与接口

- 所有路由集中在 `internal/httpapi/router.go` 注册，管理员接口使用 `/api/admin/`，
  Agent 接口使用 `/api/agent/`。
- 管理员路由必须经过 `RequireAdmin`，Agent 路由必须经过 `RequireAgent`；不得仅依赖前端隐藏入口。
- SSE 使用同源管理员 HttpOnly Cookie，不在 URL、响应或日志中暴露 Token。
- Handler 只负责解析、校验、调用 Service 和返回 JSON；事务与状态机必须留在领域层。
- 请求参数必须先校验再产生写操作；错误响应不得包含密钥、绝对工作区隐私或 SQL 细节。
- 新增接口必须同时确认鉴权主体、幂等语义、错误状态码和前端契约。

## 4. 数据库、事务与事件

- SQL 必须使用参数绑定，禁止拼接用户输入、路径、分支名或 Agent 名称。
- 多表写入、状态变更和事件落库必须处于同一事务；提交成功后才发送内存通知。
- 迁移必须可重复执行，并兼容已有 `data/app.db`，不得假设生产数据库为空。
- 状态机更新必须校验当前状态、目标状态和影响行数，禁止无条件覆盖并发结果。
- 查询列表必须有确定排序和合理分页；循环中不得产生无界查询或无界内存增长。
- 测试使用临时 SQLite 数据库，禁止连接或改写生产 `data/app.db`。

## 5. Worker、租约与并发

- Worker 必须接受 `context.Context` 或受控停止信号，并提供可重复调用的安全关闭流程。
- 轮询任务必须有固定间隔、并发上限和失败记录，禁止无间隔死循环。
- 工作包写入必须先校验租约、Agent、步骤、相对路径和工作区归属。
- 超时恢复、冻结、解冻、接管和检查点操作必须保留旧现场并记录事件。
- 幂等请求必须使用稳定幂等键；重试不得重复创建项目、工作区、报告或交付决定。
- goroutine 不得脱离应用生命周期；测试必须等待其退出，避免泄漏和偶发失败。

## 6. 文件、Git 与命令安全

- 文件访问必须通过安全路径校验，目标必须位于批准根目录内，并拒绝 `..`、符号链接逃逸和重解析点。
- Git 与外部命令使用参数数组执行，禁止将用户输入拼接为 Shell 命令。
- 每个任务和 Agent 只写入登记的分支、worktree 和 write scope。
- 同一项目的主线、Worktree、checkpoint、MR、流水线和回滚 Git 操作必须共用跨进程
  项目锁；等待锁后重新读取并核对项目 ID、目录、状态和提交身份，禁止沿用锁前快照。
- 删除 worktree、覆盖分支、回滚、发布等高风险操作必须验证归属并经过明确确认。
- 发布回滚必须持久化失败时的预期主线提交；当前 HEAD 已推进时禁止 `reset --hard`，
  只有 HEAD 等于预期提交或已等于安全提交时才允许执行或幂等恢复。
- 临时文件使用操作系统临时目录或任务专用目录，完成后清理，不写入仓库根目录。

## 7. 配置、密钥与日志

- 运行配置统一由 `internal/config` 和环境变量读取，不在源码硬编码生产路径、域名或密钥。
- Agent 密钥文件权限必须保持 `0600`；API 只返回是否已配置，不返回密钥原文。
- 日志、错误、事件和测试输出必须经过脱敏，不记录密码、Token、Cookie、API Key 或完整私密配置。
- 错误必须保留可定位的业务上下文，但不得把底层敏感信息直接暴露给调用方。

## 8. 测试文件规范

- 所有业务回归测试源文件统一放在 `server/test/testdata/`，并镜像原包路径。
- `server/test/guard/guard_test.go` 是唯一允许直接参与裸 Go 测试发现的基础设施守卫，
  用于阻止未加载归档测试的假绿结果。
- 现有测试属于同包白盒测试。执行时使用 `server/test/run.sh` 生成 Go overlay，
  将归档文件虚拟映射回原包；禁止手工复制测试到源码目录。
- overlay 运行器不得向 `cmd/` 或 `internal/` 写入临时测试，并允许不同进程使用独立配置并发执行。
- 后端全量测试命令固定为：

```bash
cd server
./test/run.sh ./...
```

- 直接执行 `go test ./...` 不会加载 `test/testdata/` 中的归档测试，并会被守卫测试拒绝，
  不能作为完整验收证据。
- 为当前任务新增的测试默认是临时测试；验证通过并记录结果后必须删除，防止污染。
- 只有用户明确要求长期保留或属于现有回归基线时，测试才可保留在 `test/testdata/`。
- 测试不得使用真实密钥、生产 Token、生产数据库或不可逆生产操作。
- 测试运行前后都必须确认 `cmd/`、`internal/` 中没有遗留 `_test.go`。

## 9. 验证、构建与部署

- 后端源码、依赖、测试或运行配置发生变化时必须执行：

```bash
cd server
./test/run.sh ./...
go build -o /tmp/wanxiang-agent-verify ./cmd/wanxiang
```

- 测试、编译或差异检查任一失败时不得部署。
- 仅修改注释、规则、函数索引或测试归档时，不替换生产二进制、不重启 PM2。
- 后端行为变化需要部署时，先备份当前二进制，再替换构建产物并执行
  `pm2 restart wanxiang-agent`；不得启动第二个监听同端口的进程。
- 部署前后必须确认 `wanxiang-agent` 为 `online`，并执行：

```bash
curl --noproxy '*' -fsS http://127.0.0.1:8088/api/health
```

- 健康检查通过只证明服务存活；鉴权、SSE、数据库和任务链路仍需按改动范围单独验收。
